# 大气污染溯源后端开发工作流（存档）

本文档沉淀从“数据接入 → 气象融合与候选筛选 → 相关性计算 →（可选）高斯羽流近似 → 融合打分 → 编排出参 → 工程化”的完整工作流，用于指导后端实现并与现有前端（React + Vite）对接。

更新时间：2025-10-16

---

## 1. 总体架构建议
- 网关/API 层：FastAPI
- 任务编排：Celery/RQ + Redis（并行计算、异步返回）
- 数据与缓存：
  - 监测/企业/气象：PostgreSQL/PostGIS 或外部 API（初期可 Mock/CSV）
  - 缓存：Redis（热点监测、风场、计算结果）
- 配置密钥：.env +（可选）Vault（AMap/外部 API Key）
- 可观测性：结构化日志、trace_id、耗时指标、失败重试与降级

---

## 2. 后端对外 API 契约

### 2.1 GET /api/config
- 作用：前端启动读取公开配置（已接入 AMap Key）
- 返回示例：
```json
{
  "amapPublicKey": "<YOUR_AMAP_KEY>",
  "features": { "dynamicPreferred": true },
  "thresholds": { "windSpeedLow": 1.5, "candidateRadiusKm": 25, "sectorHalfAngle": 11.25 },
  "scoringWeights": { "corr": 0.4, "gauss": 0.3, "angle": 0.15, "dist": 0.1, "prior": 0.05 }
}
```

### 2.2 POST /api/analyze（推荐异步）
- 入参（示例）：
```json
{
  "query": "分析广州天河站昨天的O3污染情况",
  "stationId": "Tianhe_001",
  "pollutant": "O3",
  "timeRange": { "start": "2025-10-15 00:00:00", "end": "2025-10-15 23:59:59" }
}
```
- 同步开发期可直接返回结果；生产建议返回 `jobId` 供轮询 `GET /api/analyze/{jobId}`。
- 出参结构需与前端 `src/types/api.ts` 对齐：包含 `weather_analysis / regional_analysis / voc_analysis / particulate_analysis / comprehensive_analysis` 的 `ModuleResult`，每个模块可含 `visuals`（map / timeseries / pie / bar / image）与 `anchors`。

### 2.3 内部/拆分型接口（可选对外暴露）
- POST /api/candidates：候选企业筛选
- POST /api/correlation：站点-企业（或企业站点）相关性计算
- POST /api/gaussian：高斯羽流近似估算贡献
- GET /api/timeseries：拉取监测时序

---

## 3. 数据接入与预处理
- 监测数据：站点 pollutant 的分钟/小时级；质量控制（缺测、异常、单位统一）
- 企业清单：经纬度、行业、排放强度（可缺省）、历史信用
- 气象：风速/风向/温湿度/边界层/稳定度（初期稳定度可经验估计）
- 统一时标：对齐至固定时间栅格（如小时）；明确插值/填补策略

---

## 4. 候选源筛选（气象融合规则）
- 核心规则：
  - 若风速 < 1.5 m/s：筛选站点周边 25 km 企业
  - 若风速 ≥ 1.5 m/s：筛选“上风向 ±11.25°”扇区内企业，可补充对称下风向扇区（±180°）
- 实现要点：
  - 计算球面距离（haversine）
  - 计算方位角 bearing(站点→企业) 与 angleDiff(与风向差)
  - 返回企业要素 + 距离/角度差/行业等属性
- 地图几何（输出给前端）：
  - station、enterprises、upwind_paths（可选）、sectors（中心=站点、半径=25km、半角=11.25°、方位=风向）

---

## 5. 相关性与时滞对齐
- 计算站点 vs 候选企业（或企业站点）在近 24–26 小时窗口的相关性
- 方法：Pearson/Spearman 二选一或都算；时滞对齐 0/±1 小时取最大（可扩展 ±n）
- 输出：r_max、最佳时滞 lag、显著性（可选）

---

## 6. 高斯羽流近似（可选，轻量实现）
- 目标：为每个候选源估算对受体浓度的贡献，形成“物理可解释”指标
- 简化模型：Pasquill 稳定度经验参数（σy/σz），输入 U（风速）、H（源高）、Q（排放率）、下风距 x、横向偏移 y、受体高度 z≈0
- 公式（示意）：
```
C = (Q / (2π U σy σz)) * exp(-(y^2)/(2σy^2)) * [ exp(-(z-H)^2/(2σz^2)) + exp(-(z+H)^2/(2σz^2)) ]
```
- 初期可占位（给出相对量纲），后续接入更精细的稳定度与边界层参数

---

## 7. 融合打分与排序
- 建议权重（写入 /api/config 以便动态调参）：
```
score = 0.4 * r_norm
      + 0.3 * gauss_norm
      + 0.15 * (1 - angleDiff/11.25)
      + 0.10 * (1 - distance/25km)
      + 0.05 * industry_prior
```
- 归一化策略：min-max 或分位数裁剪
- 产出：TOP K 候选企业 + 各子分解释（便于可解释性输出）

---

## 8. 出参编排（与前端契约对齐）
- weather_analysis：
  - visuals[0]（map, dynamic）：`payload` 含 map_center/station/enterprises/upwind_paths/sectors
  - anchors：如 `["风向/轨迹证据"]`
- regional_analysis：
  - visuals[0]（timeseries）：站点/企业或站点间对比；可含阈值线
- voc_analysis / particulate_analysis：
  - 行业或组分贡献图（pie/bar）
- comprehensive_analysis（仅 Markdown 文本）：
  - 结论、TOP 候选表格、证据点链接、可信度与治理建议

---

## 9. 异步编排与缓存
- 任务链/组（Celery/RQ）：
  - parse_query → fetch_data → candidates → parallel(correlation per ent, gaussian per ent) → scoring → assemble_payload
- 缓存：
  - 气象/监测按 stationId+pollutant+时间窗缓存
  - 分析结果按 query+参数hash 缓存，有效期按业务设定
- 降级：
  - 任一子任务失败置默认值、保底可视化，保证前端可渲染

---

## 10. 工程化要点
- 安全：限流、CORS、输入校验与转义、Key 管理（AMap/外部 API）
- 可观测性：结构化日志（trace_id、span）、耗时统计、异常分层
- 配置化：阈值/权重/时间窗/行业先验等统一配置与下发
