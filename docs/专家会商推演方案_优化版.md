# 多智能体专家会商推演方案（优化版）

> 面向空气质量分析报告的“事实驱动型专家会商”设计。  
> 参考本地 `D:\溯源\参考\MiroFish-main` 的多智能体仿真思路，但不照搬其社交平台仿真栈。

## 0. 本轮补充优化

根据新的目标，方案进一步调整为：

1. **引入事实层**  
   把“会商技能生成的数据表格”和“上个月污染特征与溯源分析技能生成的报告成果”作为会商的初始事实输入，而不是只把它们当作普通上下文文本。

2. **专家先读事实，再补充事实，再讨论**  
   每个专家必须先基于已有事实发表观点；如果证据不足，才调用自己被授权的对应工具补充事实数据。补充结果必须登记为新的事实，后续讨论只能引用事实编号。

3. **从 MiroFish 借鉴事实检索机制**  
   MiroFish 的 `zep_tools.py` 把图谱中的边和节点摘要整理为 `facts`，再供 `ReportAgent` 检索和写报告。这里不必引入 Zep，但应建立本地 `FactLedger`，实现类似的“事实沉淀、检索、引用、审计”能力。

4. **由“专家辩论”升级为“基于事实账本的受控会商”**  
   专家不能直接输出无法追溯的判断。所有结论、质疑、分歧和建议都必须绑定 `fact_id`、`data_id`、工具名或报告段落来源。

## 1. 评审结论

初稿已经覆盖了专家角色、讨论流程、共识输出和报告章节，方向是可行的。主要问题在于：

1. **对 MiroFish 的借鉴点偏表层**  
   MiroFish 的核心不是“几个 Agent 对话”，而是“材料建图谱、实体生成 Agent 人设、配置仿真参数、运行环境演化、报告 Agent 调用检索/采访工具生成报告”。空气质量会商应借鉴其“实体/证据/记忆/可追溯推演”机制，而不是引入 Twitter/Reddit/OASIS 这一套社交仿真。

2. **缺少事实约束，容易变成角色扮演**  
   初稿中专家可以给出“贡献约 60%/30%/10%”这类数字，但没有规定来源、计算方法、置信区间和不可下结论的边界。报告场景应强制每条观点绑定事实、数据、方法和不确定性。

3. **讨论阶段设计完整，但缺少机器可执行协议**  
   “交叉质疑、深度讨论、共识形成”需要转化为固定 JSON schema、轮次状态机、引用字段和终止条件，否则难以稳定集成到现有报告流水线。

4. **专家角色还不够贴合现有工具能力**  
   当前系统已有空气质量、气象、VOCs、PM2.5 组分、PMF、OBM、轨迹、源清单等工具。专家设计应围绕这些工具的数据能力切分，而不是仅按传统学科分工。

5. **缺少会商质量控制角色**  
   真实会商中需要主持人、反方审查员、证据审计员和报告统稿员。否则多专家输出可能只是观点堆叠，不能形成可维护的会商结论。

## 2. MiroFish 可借鉴点

已查阅 MiroFish README 和关键源码，建议借鉴以下机制：

| MiroFish 机制 | 源码体现 | 对空气质量会商的改造 |
|---|---|---|
| 种子材料建图谱 | `GraphBuilderService` 使用 Zep 构建图谱 | 把阶段 5 的分析结果、数据表、图表、指标摘要构造成“会商证据包” |
| 实体生成 Agent Profile | `OasisProfileGenerator` 从实体生成 profile/persona | 不自动生成社会人物，而是生成专家卡片、数据权限、方法偏好和证据要求 |
| LLM 自动生成仿真配置 | `SimulationConfigGenerator` 生成时间、事件、Agent 活跃配置 | 自动生成会商议题、轮次、专家参与顺序、压力测试场景 |
| 仿真后报告 Agent | `ReportAgent` 通过 `insight_forge`、`panorama_search`、`interview_agents` 写报告 | 会商主持人通过“证据检索、观点追问、反方审查”工具生成结论 |
| 过程日志 | `ReportLogger` 写 `agent_log.jsonl` | 保存专家观点、质疑、证据引用、共识计算过程，支持审计和复盘 |
| 事实检索 | `ZepToolsService.search_graph/insight_forge/panorama_search` 返回 facts | 建立本地 FactLedger，支持按污染物、城市、时间、专家、事实类型检索 |
| 时序记忆更新 | `zep_graph_memory_updater.py` 将仿真过程写回图谱 | 将专家补充工具结果写回事实账本，形成“初始事实 + 补证事实 + 结论事实”链路 |

不建议照搬：

- 不建议引入 OASIS/Twitter/Reddit 双平台仿真。空气质量会商不是社会舆论扩散问题，使用社交平台模型会增加复杂度且解释性不足。
- 不建议把“自由交互”作为首版核心。报告生成需要稳定、可复现、可审计，首版应采用受控轮次和结构化输出。
- 不建议依赖“涌现结论”。业务报告需要每个结论可回溯到数据、方法和专家判断。

## 3. 优化后的系统定位

新增阶段建议命名为：

```text
阶段5.2：证据驱动型专家会商推演
```

它不是替代阶段 5 深度分析，而是在阶段 5 结果之上做：

- 多专家独立审阅
- 关键议题提炼
- 交叉质疑和缺口补证
- 假设/压力测试
- 共识、分歧和不确定性输出
- 生成可直接进入 `report.qmd` 的会商结论

更准确地说，阶段 5.2 的输入分为两类：

1. **种子事实**  
   来自会商技能已经生成的数据表格，以及“上个月污染特征与溯源分析”技能已经生成的报告成果。这些内容代表系统在会商前已经确认或初步分析过的事实。

2. **补充事实**  
   会商过程中，专家发现证据不足或需要交叉验证时，调用自己配置的工具获取的新数据、新分析结果或新模型输出。

会商最终不是“谁说服谁”，而是形成一条可追溯链路：

```text
种子事实 → 专家观点 → 专家补证 → 交叉质疑 → 共识/分歧 → 报告结论
```

## 4. 推荐架构

```text
阶段5深度分析结果 + 会商技能数据表格 + 上月污染特征与溯源报告成果
  ├─ 指标摘要：污染物、城市、时段、同比环比、排名
  ├─ 会商表格：Excel/CSV/Markdown表格、重点城市清单、污染过程清单
  ├─ 报告成果：已生成章节、阶段性结论、图表说明、已有管控建议
  ├─ 数据证据：data_id、表格、图表、工具输出
  ├─ 模型结果：PMF、OBM/OFP、轨迹、源清单、预测
  └─ 已生成分析文本
          ↓
事实账本构建 FactLedger
          ↓
证据包构建 EvidencePack（从 FactLedger 按议题抽取）
          ↓
会商配置生成 DeliberationConfig
  ├─ 专家清单
  ├─ 议题清单
  ├─ 必查证据
  ├─ 轮次策略
  └─ 压力测试场景
          ↓
专家独立分析（并行）
  ├─ 读取相关事实
  ├─ 标注支持/反驳/不足
  └─ 必要时调用授权工具补充事实
          ↓
主持人汇总议题和冲突点
          ↓
交叉质疑 + 缺口补证
          ↓
反方审查 + 一致性校验
          ↓
共识生成 + 分歧保留
          ↓
会商结论 JSON + report.qmd 章节
```

## 4.1 事实账本 FactLedger

建议新增 `FactLedger`，专门管理会商事实。它对应 MiroFish 中“图谱 facts + ReportAgent 检索工具”的轻量替代方案。

### 4.1.1 事实来源

| 来源 | 示例 | 进入方式 | 可信度初值 |
|---|---|---|---|
| 会商技能数据表格 | `tables/monthly_pollution_summary.xlsx`、TOP城市表、污染过程表 | 表格解析为结构化事实 | 高 |
| 上月污染特征与溯源分析报告 | 已生成 Markdown/Quarto 章节、阶段性分析结论 | 段落切分 + 结论抽取 | 中高 |
| 已有 data_id 数据 | `air_quality:v1:xxx`、`weather:v1:xxx` | 从 `DataContextManager` 加载摘要 | 高 |
| 已有模型结果 | PMF、OBM/OFP、轨迹、重构、预测 | 工具结果转事实 | 中高 |
| 专家补充工具结果 | 会商过程中调用气象、组分、轨迹等工具 | 新结果登记为补充事实 | 按数据质量评估 |
| 人工输入/用户补充 | 用户上传说明、人工会商意见 | 标注为人工事实 | 需复核 |

### 4.1.3 ClaimRecord

专家观点必须从事实生成，不能直接进入最终结论。

```json
{
  "claim_id": "claim_meteorology_001",
  "expert_id": "meteorology_expert",
  "claim": "4月中旬PM2.5污染过程受静稳和弱风条件放大。",
  "claim_type": "causal_interpretation",
  "supporting_facts": ["fact_weather_010", "fact_air_022"],
  "contradicting_facts": [],
  "missing_facts": ["边界层高度连续观测"],
  "confidence": 0.76,
  "status": "candidate"
}
```

### 4.1.4 事实使用规则

- 专家发言必须引用 `fact_id` 或说明需要补证。
- 工具调用结果必须先登记为 `FactRecord`，再参与讨论。
- 报告正文只能引用 `accepted` 状态的结论事实。
- 无事实支撑的观点进入 `unsupported_claims`，不得写入正式报告。
- 事实冲突不能删除，应标记 `conflict_group_id`，交由主持人和反方审查员处理。

## 5. 专家角色优化

首版建议使用 7 个角色，其中 5 个领域专家、2 个质量控制角色。每个专家还要配置“事实读取范围”和“补证工具白名单”。

| 角色 | 代号 | 核心职责 | 初始事实范围 | 补证工具白名单 |
|---|---|---|---|---|
| 监测数据专家 | `monitoring_expert` | 识别污染过程、城市排名、同比环比、异常点 | 会商表格、监测统计、报告污染过程章节 | `get_air_quality`, `calculate_iaqi`, `load_data_from_memory` |
| 气象扩散专家 | `meteorology_expert` | 判断扩散、清除、静稳、边界层和风场影响 | 报告气象章节、气象表格、污染过程时段 | `get_weather_data`, `meteorological_trajectory_analysis`, `load_data_from_memory` |
| 组分化学专家 | `chemistry_expert` | 判断二次生成、组分结构、O3/VOCs/NOx 机制 | 组分表格、VOCs/PM2.5章节、报告机制判断 | `get_vocs_data`, `get_pm25_ionic`, `get_pm25_carbon`, `get_pm25_crustal`, `calculate_obm_ofp`, `calculate_obm_full_chemistry`, `calculate_reconstruction`, `calculate_carbon`, `calculate_soluble`, `calculate_crustal`, `load_data_from_memory` |
| 来源解析专家 | `source_apportionment_expert` | PMF/源解析、源类贡献、工业/交通/扬尘判断 | PMF结果、源解析章节、企业/源类表 | `calculate_pm_pmf`, `calculate_vocs_pmf`, `calculate_pmf`, `calculate_reconstruction`, `load_data_from_memory` |
| 区域传输专家 | `transport_expert` | 本地累积与外来传输识别 | 轨迹章节、上风向企业表、污染过程时段 | `meteorological_trajectory_analysis`, `analyze_upwind_enterprises`, `analyze_trajectory_sources`, `get_weather_data`, `load_data_from_memory` |
| 反方审查员 | `skeptic_reviewer` | 挑出证据不足、过度归因、数字不可信 | 全部事实账本和候选观点 | `load_data_from_memory`，原则上不新增业务数据 |
| 主持统稿员 | `moderator_writer` | 控制轮次、生成共识、输出报告章节 | 全部事实、观点、审查意见 | `load_data_from_memory`，必要时请求专家补证 |

角色提示词不应只写“你是谁”，还要写清：

- 可使用的数据范围
- 必须输出的字段
- 禁止无证据推断
- 何时必须声明“不足以下结论”
- 如何标注置信度
- 如何质疑其他专家

### 6.2.1 专家补证规则

专家只有在以下情况下调用工具：

- 种子事实缺少该专家判断所需的关键变量。
- 其他专家观点与已有事实冲突，需要补证验证。
- 需要把报告中的文字结论回查为原始数据。
- 主持人或反方审查员明确要求补证。

专家不应为“丰富内容”随意调用工具。每次工具调用都要写明：

- 要验证的问题
- 已有事实缺口
- 调用工具名和参数
- 生成的新事实编号
- 新事实是否改变原观点

```json
      "needed_evidence": "轨迹聚类和上风向同期浓度对比"
    }
  ],
  "forbidden_claims": [
    {
      "claim": "气象贡献60%、排放贡献30%、传输贡献10%",
      "reason": "当前证据包没有定量归因模型支撑，不能写入正式报告"
    }
  ]
}
```

## 7. 推演轮次优化

建议首版固定为 5 轮，避免无限讨论。

| 轮次 | 名称 | 输入 | 输出 |
|---|---|---|---|
| R0 | 会商配置 | EvidencePack | 专家清单、议题、必查证据 |
| R1 | 事实读取 | FactLedger + EvidencePack | 每位专家的相关事实子集 |
| R2 | 独立分析与补证 | 事实子集 + 专家工具白名单 | `expert_analyses.json`、新增事实 |
| R3 | 观点对齐 | 专家分析结果 | 议题矩阵、共识候选、冲突点 |
| R4 | 交叉质疑 | 冲突点 + 专家问题 | `discussion_log.jsonl`、必要补证事实 |
| R5 | 反方审查 | 共识草案 | 风险、证据缺口、禁写结论 |
| R6 | 统稿输出 | 修订后共识 | `consensus.json`、会商章节 Markdown |

首版不建议做超过 2 次循环补证。补证成本高，且报告生成链路会变慢。可以设置：

- 每个专家最多 3 条核心观点
- 每条观点最多 5 个证据引用
- 每个冲突点最多 2 轮追问
- 无证据的新结论直接进入 `forbidden_claims`
- 每个专家最多 3 次补证工具调用，除非用户明确要求深度推演
- 补证结果必须进入 FactLedger，不能只停留在专家回答文本中

## 8. 共识度计算

不要让 LLM 自称“共识度 0.85”。建议用可解释规则计算：

```text
consensus_score =
  0.35 * 专家支持率
+ 0.30 * 证据充分度
+ 0.20 * 反方审查通过率
+ 0.15 * 数据质量系数
```

字段解释：

- 专家支持率：支持该结论的专家数 / 参与专家数
- 证据充分度：必需证据是否覆盖，分 strong/medium/weak
- 反方审查通过率：反方指出的问题中已解决比例
- 数据质量系数：数据完整性、时间覆盖、方法适用性

报告中只建议写：

- `高共识`
- `较高共识`
- `存在分歧`
- `证据不足`

不建议在正文写精确小数，除非用户确实需要审计表。

## 9. 与现有系统集成

当前项目已有 `call_sub_agent`、专家模式工具、`DataContextManager` 和模板报告生成流程。建议第一阶段不新增复杂 Agent 框架，而是新增一个会商编排服务：

```text

  ├─ expert_registry.py          # 专家定义与提示词路径

  ├─ fact_ingestor.py            # 从会商表格、上月报告、data_id导入事实
  ├─ evidence_builder.py         # 汇总阶段5数据和 data_id
```

### 9.1 输入适配建议

对会商技能表格：

- Excel/CSV 表格按 sheet/table/row/cell 定位。
- 每一行可转为一个或多个 `FactRecord`。
- 对“TOP3城市”“污染过程”“同比环比”“主要污染物”等高价值表格，额外生成 `derived_metric` 类型事实。

对“上个月污染特征与溯源分析”报告成果：

- 章节标题、段落、表格、图注分别抽取。
- 明确数值和判断句转为 `report_finding`。
- 图表说明转为 `chart_interpretation`。
- 管控建议转为 `recommendation_seed`，后续由专家审查是否保留。

对现有 `data_id`：

- 不复制全量原始数据进 prompt。
- 通过 `DataContextManager` 读取摘要、字段、时间范围和统计指标。
- 专家需要核查时，通过 `load_data_from_memory` 或对应业务工具获取局部数据。

## 10. 报告章节建议

正式报告中不宜展示过长“聊天记录”。建议展示会商结论、关键分歧和证据链。

```markdown
# 九、专家会商结论

## 9.1 会商依据

本次会商基于空气质量监测、气象、组分、源解析和轨迹分析结果开展。会商重点围绕污染过程识别、成因判定、区域传输影响和下月风险研判。

## 9.2 主要共识

| 结论 | 共识等级 | 主要证据 | 不确定性 |
|---|---|---|---|
| 本月PM2.5污染过程主要受不利扩散条件与二次生成增强共同影响 | 较高共识 | 监测浓度、气象要素、离子组分 | 气象和二次生成贡献比例不能精确拆分 |

## 9.3 分因素判断

| 因素 | 判断 | 证据 | 会商意见 |
|---|---|---|---|
| 气象扩散 | 不利 | 风速、降水、静稳过程 | 对污染累积有放大作用 |
| 二次生成 | 增强 | 硝酸盐/硫酸盐占比 | 是细颗粒物升高的重要机制 |
| 区域传输 | 存在但贡献待定 | 后向轨迹、上风向浓度 | 需要进一步量化 |

## 9.4 分歧与后续补证

| 分歧点 | 当前判断 | 后续建议 |
|---|---|---|
| 区域传输贡献比例 | 存在分歧 | 增加轨迹聚类、PSCF/CWT或上风向城市同步浓度对比 |

## 9.5 管控建议

1. 针对静稳过程前后，提前开展重点源排放调度和扬尘控制。
2. 对二次生成风险较高时段，加强NOx、VOCs和氨排放协同管控。
3. 对区域传输风险过程，强化上风向城市协同研判和预警联动。
```

## 11. 实施计划

### 第一阶段：轻量可用版（1-2周）

- 新增专家提示词硬约束和 JSON schema
- 新增 FactLedger 和 FactIngestor
- 从会商表格、上月分析报告、阶段 5 输出构建事实账本
- 基于事实账本构建 EvidencePack
- 并行调用 5 个领域专家
- 允许专家在工具白名单内有限补证
- 主持人汇总共识、分歧、禁写结论
- 输出 `consensus.json` 和报告章节 Markdown

验收标准：

- 每条会商结论都有 `fact_id` / `data_id` / 报告段落引用
- 专家补充事实会写入 `fact_ledger.jsonl`
- 没有证据的定量贡献比例不会进入正文
- 能稳定插入 `report.qmd`

### 第二阶段：审查增强版（1周）

- 增加 `skeptic_reviewer`
- 增加结论一致性检查
- 增加 `forbidden_claims.json`
- 增加会商过程日志

验收标准：

- 能识别“过度归因”“无依据比例”“数据缺口”
- 能输出正式报告可用和不可用结论清单

### 第三阶段：推演增强版（2周）

- 增加情景注入：不利气象、排放增加、区域传输、臭氧高风险
- 增加下月风险研判对比
- 增加会商结论版本对比

验收标准：

- 同一证据包可生成基准情景和压力情景结论
- 报告能说明“若出现某类条件，风险如何变化”

## 12. 关键工程注意事项

1. **优先用现有 `call_sub_agent` 和专家工具**  
   不要首版就引入 MiroFish 的 OASIS/Zep 全栈。当前项目已有数据查询和分析工具，直接复用成本最低。

2. **所有专家输出必须是 JSON，可再渲染为 Markdown**  
   先结构化、后写作，避免会商过程不可控。

3. **贡献比例必须有模型来源**  
   没有 CMAQ、源解析定量、观测约束模型或明确计算方法时，不写“60%/30%/10%”。

4. **保留分歧比强行共识更重要**  
   空气质量成因复杂，报告可信度来自证据边界，而不是所有专家一致。

5. **日志必须可追溯**  
   建议仿照 MiroFish `ReportLogger`，保存 `agent_log.jsonl`，记录每轮输入、输出、证据引用和审查意见。

## 13. 优化后的一句话方案

在现有报告生成流程中新增一个“事实驱动型专家会商编排层”：它先把会商技能生成的数据表格、上个月污染特征与溯源分析报告成果、阶段 5 数据和模型结果统一登记为 FactLedger，再让多个专家基于事实独立分析并按工具白名单补证，最后由主持人与反方审查员围绕事实链形成可审计的共识、分歧、禁写结论和正式报告章节。
